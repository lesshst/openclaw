# 源码版接管与自动同步部署

## 背景

- 原先本机默认 `openclaw` 命令实际指向全局安装版，源码仓只能通过 `pnpm openclaw` 间接使用。
- 原先后台实际运行中的 gateway 也是安装版服务，源码版脚本虽然存在，但没有真正接管默认运行链路。
- 目标是让源码版成为日常主版本，同时保留 fork 中的 WhatsApp 代理补丁，并把上游同步、构建和重启流程固化为每日自动任务。

## 变更目标

1. 让默认 `openclaw` 命令直接落到源码版。
2. 让后台 gateway 由源码版接管，而不是继续使用安装版服务。
3. 避免在日常开发仓上做无人值守 `merge`，降低自动同步对本地未提交改动的干扰。
4. 让每日自动同步在代理网络环境下也能稳定执行。

## 主要调整

### 1. 运行仓与开发仓分离

- 不直接在日常开发仓上做自动 `fetch`、`merge`、`build` 和 `restart`。
- 单独准备一个“运行仓”作为实际服务入口，用来承接：
  - 上游同步
  - 依赖安装
  - CLI 构建
  - 控制台 UI 构建
  - gateway 重启与验活
- 这样可以避免开发仓存在未提交改动时，被自动任务打断或污染。

### 2. 源码版接管默认 CLI

- 将默认 `openclaw` 命令切换为源码运行仓版本，而不是继续使用 npm 全局安装版。
- 切换完成后，日常直接执行 `openclaw --version`，返回的是源码版版本号。
- 安装版不再作为主入口，只保留源码版作为统一命令入口。

### 3. 源码版 gateway 接管后台服务

- 停用原先的安装版 gateway 服务。
- 由源码版 `ai.openclaw.gateway-src` 接管 `18789` 端口。
- gateway 状态判断不再只看“端口是否监听”，而是联合校验：
  - `launchctl` label
  - 实际进程 `pid`
  - 端口监听 `pid`
  - 进程工作目录
  - HTTP 健康探针

这一步的意义是：可以明确确认“当前运行的是源码版服务”，而不是只看到端口被某个旧进程占住。

### 4. 补强源码版 gateway 脚本可靠性

- `scripts/gw-status.sh` 修复为“按所有权判定状态”，避免把其他进程占用端口误判成源码版正常。
- `scripts/gw-start.sh` 延长就绪等待时间，避免服务已经启动但脚本过早报失败。
- `scripts/gw-common.sh` 用于集中解析仓库根目录、服务 label、日志、端口和 Node 路径，减少脚本里的硬编码。
- 启停逻辑补上竞争服务检查，避免先停自己的服务、再因为官方服务占端口而启动失败。

### 5. 固化每日自动同步与构建流水线

- 新增每日自动任务 `ai.openclaw.src-sync`。
- 默认调度时间为每天 `05:30`。
- 自动任务的执行链路固定为：
  1. `fetch upstream/main`
  2. 尝试 `merge upstream/main`
  3. `pnpm install`
  4. `pnpm build`
  5. `pnpm ui:build`
  6. `scripts/gw-restart.sh`
  7. `scripts/gw-status.sh`

同时加入了几条保护规则：

- 运行仓工作树不干净时，直接拒绝自动同步。
- 如果存在未完成的 merge 状态，先尝试中止旧 merge。
- 如果新的 merge 冲突，自动中止并退出，不继续构建和重启。

这意味着自动任务只会在“同步成功且构建成功”的前提下接管运行，不会把一个半失败状态推进到线上。

### 6. 补齐后台任务代理能力

- 手工在终端执行同步时，shell 自身会带上 `HTTP_PROXY` / `HTTPS_PROXY`。
- 但 `launchd` 后台任务默认不继承这些代理变量，因此首次后台实跑时卡在 `git fetch upstream`。
- 为解决这个问题，自动同步脚本改成了两级策略：
  - 优先使用已有的代理环境变量
  - 如果后台环境里没有代理变量，则自动读取 macOS 系统代理配置

同时补充：

- `GIT_TERMINAL_PROMPT=0`
- `GIT_ASKPASS=/usr/bin/true`

这样可以避免后台任务因为认证提示或网络代理缺失而长时间挂起。

## 实际验证结果

本次调整完成后，已验证以下结果：

- 默认 `openclaw` 命令已经切到源码版。
- 源码版 gateway 已经接管 `18789` 端口。
- `scripts/gw-status.sh` 可正确判断源码版服务归属并通过健康检查。
- `ai.openclaw.src-sync` 已成功装载到 `launchd`。
- 手动触发一次后台同步任务后，完整走通了：
  - 代理补全
  - `fetch upstream/main`
  - `merge upstream/main`
  - `pnpm install`
  - `pnpm build`
  - `pnpm ui:build`
  - gateway 重启
  - gateway 验活
- 最终后台任务退出码为 `0`，日志中出现 `Source sync completed successfully`。

## 当前效果

调整完成后，当前使用方式变成：

- 平时直接使用 `openclaw`，默认就是源码版。
- 后台常驻服务默认使用源码版 gateway。
- 每天清晨会自动同步上游、自动构建、自动重启并做健康检查。
- WhatsApp 代理补丁继续保留在源码版链路中，不再依赖安装版。

## 后续建议

- 运行仓现在主要对齐的是 `upstream/main`，适合做稳定运行入口。
- 如果后续还需要把个人 fork 的 `origin/main` 也一起保持同步，可以再单独补一条“运行仓与个人 fork 对齐”的维护流程。
- 这类自动化依赖本机代理环境，若后续系统代理端口调整，需要同步检查 gateway 和自动同步任务的代理行为是否仍然一致。
