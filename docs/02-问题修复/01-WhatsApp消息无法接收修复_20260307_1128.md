# WhatsApp 消息无法接收修复记录

## 基本信息

- 问题描述：通过手机 WhatsApp 发送消息给 OpenClaw，OpenClaw 无法收到消息。
- 处理时间：2026-03-07 10:45 至 11:27（Asia/Shanghai）
- 影响范围：OpenClaw 的 WhatsApp 入站和出站能力均受影响。
- 处理目标：
  - 恢复当前环境的 WhatsApp 消息接收能力。
  - 找出根因，避免服务切换后问题再次出现。
  - 将修复沉淀到源码和当前实际运行版本中。

## 现象与初步判断

排查开始时，配置表面上是正常的：

- `channels.whatsapp.enabled=true`
- 账号 `default` 已配置、已 linked
- 本地凭据目录 `/Users/lizhibo/.openclaw/credentials/whatsapp/default` 存在，且包含 `creds.json` 等文件

但运行状态异常：

- `openclaw channels status --probe` 显示：
  - `enabled, configured, linked, stopped, disconnected`
- `gateway.log` 持续出现：
  - `starting provider`
  - `auto-restart attempt ...`
- `gateway.err.log` 持续出现：
  - `WebSocket Error (Opening handshake has timed out)`

因此可以先排除以下方向：

- 不是 `allowFrom`、群策略、消息路由策略导致的过滤问题。
- 不是凭据目录缺失导致的“未登录”问题。
- 不是 webhook / inbound parser / agent 层没有处理消息的问题。

初步结论：问题发生在 WhatsApp Web 连接建立阶段，通道根本没有保持在线。

## 关键排查过程

### 1. 确认不是业务层逻辑问题

查看配置、扩展入口和运行日志后，确认异常发生在 WhatsApp provider 启动时，而不是收到消息之后：

- 配置文件：`/Users/lizhibo/.openclaw/openclaw.json`
- 源码入口：`extensions/whatsapp/src/channel.ts`
- 运行日志：
  - `/Users/lizhibo/.openclaw/logs/gateway.log`
  - `/Users/lizhibo/.openclaw/logs/gateway.err.log`

日志显示 provider 每次启动后都在 WebSocket opening handshake 阶段超时退出，然后进入自动重试。

### 2. 验证网络不是完全不通

对 `https://web.whatsapp.com` 做 HTTPS 连通性验证时，请求可以返回 `HTTP/2 200`。这说明：

- 机器不是完全无法访问 WhatsApp。
- 常规 HTTPS 请求是通的。

因此问题进一步收敛到：

- WhatsApp Web 的实时 WebSocket 链路。
- 或该链路在当前代理/网络环境下没有正确走代理。

### 3. 直接验证 WebSocket 与代理行为

使用 `ws` 直接连接以下地址：

- `wss://web.whatsapp.com/ws/chat`
- `wss://web-fallback.whatsapp.com/ws/chat`
- `wss://web.whatsapp.com:5222/ws/chat`

得到两组关键结果：

- 不显式传代理时：
  - `Opening handshake has timed out`
- 显式传 `HttpsProxyAgent('http://127.0.0.1:7897')` 时：
  - 返回 `101 Switching Protocols`
  - WebSocket 能正常 `OPEN`

这一步说明根因与“WhatsApp Web 连接未正确使用代理”高度相关。

### 4. 识别“源码版本”和“实际运行版本”不一致

进一步检查后发现，当前系统中同时存在两套 OpenClaw 运行体：

- 全局安装版：
  - 运行中的默认服务 `ai.openclaw.gateway`
  - 版本：`openclaw 2026.3.2`
- 本地源码版：
  - 工作区：`/Users/lizhibo/.openclaw/workspace/tmp-openclaw-src`
  - 源码版本：`2026.2.27`

对两者做对比后发现：

- 源码版 `src/web/session.ts` 已经带有 WhatsApp 代理逻辑：
  - 会创建 `HttpsProxyAgent`
  - 会把 `agent` / `fetchAgent` 传给 Baileys `makeWASocket`
- 全局安装版的实际 bundle 中，`createWaSocket()` 没有把代理传给 Baileys

这解释了为什么：

- 配置里明明有代理环境变量
- 机器普通 HTTPS 也能访问 WhatsApp
- 但运行中的 WhatsApp provider 仍然持续握手超时

根因不是“没有代理配置”，而是“运行中的实际版本没有把代理真正用到 WhatsApp Web socket 上”。

### 5. 识别一个额外兼容性问题

源码版里虽然已经有代理逻辑，但它最初只读取：

- `HTTPS_PROXY`
- `HTTP_PROXY`

而当前 shell 中也存在：

- `https_proxy`
- `http_proxy`

为了避免今后出现“终端里代理可用，但 WhatsApp 不读取小写环境变量”的边缘问题，补充了大小写兼容。

## 根因结论

本次问题的根因分为两层：

### 根因 1：实际运行的全局安装版未将代理传入 WhatsApp Web 连接

全局安装版 `openclaw 2026.3.2` 中，WhatsApp `createWaSocket()` 对 Baileys 的 `makeWASocket()` 调用缺少：

- `agent`
- `fetchAgent`

在当前网络环境下，未走代理的 `wss://web.whatsapp.com/ws/chat` 会直接在 opening handshake 阶段超时，导致 provider 不断重启。

### 根因 2：源码中代理环境变量读取不够鲁棒

源码版已有代理支持，但只读取大写代理环境变量，未兼容小写：

- 需要同时支持 `HTTPS_PROXY / HTTP_PROXY`
- 也需要支持 `https_proxy / http_proxy`

否则不同启动方式下仍有概率出现代理未生效的问题。

## 修复过程

### 第一阶段：先恢复可用性

为了尽快恢复消息收发，先切换到本地源码版网关服务：

- 停掉默认服务：`ai.openclaw.gateway`
- 启动源码版服务：`ai.openclaw.gateway-src`

切换后复查：

- `openclaw channels status --probe`
- 结果恢复为：
  - `WhatsApp default: enabled, configured, linked, running, connected`

这一步证明源码版路径确实能正常连上 WhatsApp Web。

### 第二阶段：修复源码中的代理环境变量兼容

修改源码文件：

- `src/web/session.ts`
- `src/web/session.test.ts`

修复内容：

- `resolveWhatsAppProxyAgent()` 同时读取：
  - `HTTPS_PROXY`
  - `https_proxy`
  - `HTTP_PROXY`
  - `http_proxy`
- 增加回归测试，验证只有小写代理环境变量存在时，仍会把代理传给 `makeWASocket()`

验证结果：

- 执行 `pnpm exec vitest run src/web/session.test.ts`
- 结果：`9 tests passed`

### 第三阶段：修复全局安装版，恢复默认运行方式

为了避免之后切回默认服务又复发，对全局安装版的实际 bundle 进行了修补。

补丁内容：

- 引入 `HttpsProxyAgent`
- 增加 `resolveWhatsAppProxyAgent()`
- 在 `makeWASocket()` 调用中注入：
  - `agent: proxyAgent`
  - `fetchAgent: proxyAgent`
- 代理环境变量兼容大小写

实际修补的全局安装版文件如下：

- `/Users/lizhibo/.nvm/versions/node/v24.12.0/lib/node_modules/openclaw/dist/session-3vIgvgDO.js`
- `/Users/lizhibo/.nvm/versions/node/v24.12.0/lib/node_modules/openclaw/dist/session-B5yqQdWr.js`
- `/Users/lizhibo/.nvm/versions/node/v24.12.0/lib/node_modules/openclaw/dist/session-DHZIVqkT.js`
- `/Users/lizhibo/.nvm/versions/node/v24.12.0/lib/node_modules/openclaw/dist/session-DXmcD6Pj.js`

说明：

- 全局安装版里存在多份编译后的 `createWaSocket()` 副本。
- 仅修一份不足以覆盖 CLI、登录流程、网关 bundle 等全部入口。
- 需要将实际会被加载到的多个 session bundle 一并补齐。

### 第四阶段：切回默认服务并验证稳定性

在全局安装版修补完成后：

- 停止源码版服务 `ai.openclaw.gateway-src`
- 恢复默认服务 `ai.openclaw.gateway`
- 对默认服务执行重启与复查

最终验证结果：

- 默认服务在线
- `openclaw channels status --probe` 连续两次返回：
  - `WhatsApp default: enabled, configured, linked, running, connected`

## 结论

本次问题本质上不是消息接收逻辑错误，而是 WhatsApp Web 连接在当前网络环境下没有正确走代理，导致通道长期离线。修复完成后，默认服务和源码服务两条运行路径都已具备代理能力，WhatsApp 通道已恢复为正常连接状态。
