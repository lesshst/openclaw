# 多账号恢复与自动同步修复及 legacy auth.json 清理

## 1. 问题背景

在给 `main` agent 添加第二个 ChatGPT / OpenAI Codex 账号，并同步到其他 agent 后，现场出现了配置回退：

- `main` 的 `auth-profiles.json` 从双账号又变回单账号
- `mc-gateway-520239c5-b9b5-475e-b460-bc9f59571f95` 的 `auth-profiles.json` 也被压回单账号
- `project-task` 暂时仍保留双账号，因此现场状态不一致

用户观察到的现象是：

- `/Users/lizhibo/.openclaw/agents/main/agent/auth-profiles.json`
- `/Users/lizhibo/.openclaw/agents/mc-gateway-520239c5-b9b5-475e-b460-bc9f59571f95/agent/auth-profiles.json`

这两个文件中只剩一个 `openai-codex:default`。

## 2. 现场影响

- `main` 和 `mc-gateway` 丢失第二个 ChatGPT 账号 profile
- 多 agent 的账号配置不一致，后续路由和账号选择不可预测
- 运行中的 gateway 会继续把单账号状态扩散到其他 agent
- `main` 还保留旧格式 `auth.json`，存在再次把多账号现场“拉回单账号”的风险

## 3. 根因分析

### 3.1 跨 agent OAuth 自动同步逻辑会覆盖多账号配置

运行中的源码版 gateway 在：

- `/Users/lizhibo/.openclaw/runtime/openclaw-src-live/src/agents/auth-profiles/oauth.ts`

中有两条“从 `main` agent 继承更新凭据”的路径：

1. `adoptNewerMainOAuthCredential(...)`
2. OpenAI Codex refresh 失败后的 fallback 到 `main`

这两条路径原先只看“同 provider 下是否有更新的 OAuth 凭据”，没有识别“当前 agent 其实是多 profile provider”。  
结果是，只要 `main` 上某个 provider 当时只剩单个 `openai-codex:default`，其他 agent 就可能被它覆盖并落盘。

对应现场日志里曾多次出现：

- `adopted newer OAuth credentials from main agent`
- `inherited fresh OAuth credentials from main agent`

### 3.2 `main` 的 legacy `auth.json` 仍保留单账号残影

`main` agent 还存在旧格式认证文件：

- `/Users/lizhibo/.openclaw/agents/main/agent/auth.json`

其中只有单个 `openai-codex` OAuth 凭据。  
这会让某些认证发现/迁移路径继续把 `main` 视为单账号来源，即使 `auth-profiles.json` 已经恢复成双账号，仍有再次退化的可能。

## 4. 修复目标

本次修复分三步：

1. 先把 `main` 和 `mc-gateway` 恢复成与 `project-task` 一致的双账号状态
2. 修掉跨 agent OAuth 自动同步逻辑，避免多账号 provider 再次被压回单账号
3. 清理 legacy `auth.json` 残影，封住再次退化的入口

## 5. 实际处理过程

### 5.1 恢复双账号配置

以仍保持双账号的 `project-task` 为对照，把以下两个 agent 恢复为双账号：

- `/Users/lizhibo/.openclaw/agents/main/agent/auth-profiles.json`
- `/Users/lizhibo/.openclaw/agents/mc-gateway-520239c5-b9b5-475e-b460-bc9f59571f95/agent/auth-profiles.json`

恢复后的两个 profile 为：

- `openai-codex:default`
- `openai-codex:lizhibo360106773@gmail.com`

并统一 `openai-codex` 的顺序为：

1. `openai-codex:default`
2. `openai-codex:lizhibo360106773@gmail.com`

现场备份文件：

- `/Users/lizhibo/.openclaw/agents/main/agent/auth-profiles.json.restore-dual-20260307-2048.bak`
- `/Users/lizhibo/.openclaw/agents/mc-gateway-520239c5-b9b5-475e-b460-bc9f59571f95/agent/auth-profiles.json.restore-dual-20260307-2048.bak`

### 5.2 修复自动同步逻辑

修改文件：

- `/Users/lizhibo/.openclaw/runtime/openclaw-src-live/src/agents/auth-profiles/oauth.ts`

处理方式：

- 新增 `shouldAutoSyncMainOAuthCredential(...)`
- 只要某个 provider 在当前 agent 可见到多个 profile，就禁止跨 agent 自动采用 `main` 的 OAuth 凭据
- 同时收紧 refresh 失败后的 fallback 路径，避免多 profile provider 在失败恢复时被单 profile 覆盖

这样可以保证：

- 单账号 provider 仍可沿用原来的“从 `main` 继承更新凭据”能力
- 多账号 provider 不再被跨 agent 自动同步逻辑错误压平

### 5.3 补回归测试

新增/补充的测试文件：

- `/Users/lizhibo/.openclaw/runtime/openclaw-src-live/src/agents/auth-profiles/oauth.fallback-to-main-agent.test.ts`
- `/Users/lizhibo/.openclaw/runtime/openclaw-src-live/src/agents/auth-profiles/oauth.openai-codex-refresh-fallback.test.ts`

覆盖点包括：

- 多 profile provider 时不再从 `main` 自动 adopt
- refresh fallback 不再把多账号 provider 压回单账号

### 5.4 清理 legacy auth.json 路径

修改文件：

- `/Users/lizhibo/.openclaw/runtime/openclaw-src-live/src/agents/pi-model-discovery.ts`

处理方式：

- 调整 legacy `auth.json` 清理逻辑
- 只要某个 provider 已经由 `auth-profiles.json` 提供 runtime 凭据，就清掉 `auth.json` 中同 provider 的旧条目

现场处置：

- 备份 `/Users/lizhibo/.openclaw/agents/main/agent/auth.json`
- 删除旧文件，避免单账号残影继续参与认证发现

备份文件：

- `/Users/lizhibo/.openclaw/agents/main/agent/auth.json.legacy-backup-20260307-2048.bak`

相关测试：

- `/Users/lizhibo/.openclaw/runtime/openclaw-src-live/src/agents/pi-model-discovery.auth.test.ts`
- `/Users/lizhibo/.openclaw/runtime/openclaw-src-live/src/agents/pi-auth-json.test.ts`

## 8. 后续建议

- 如果未来再为 agent 增加第三个账号，优先在 `auth-profiles.json` 体系内完成，不再依赖 legacy `auth.json`
- 对 `auth-profiles.json` 建议补文件权限收紧，例如 `chmod 600`
- 如果后续要把这些修复合并回正式发布版本，需要同步到非源码版 gateway 的安装产物，避免升级后被覆盖
