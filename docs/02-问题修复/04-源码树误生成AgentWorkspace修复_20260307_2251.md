# 源码树误生成 Agent Workspace 修复

## 1. 问题现象

用户在运行源码目录下发现了一个不应属于源码的目录：

- `/Users/lizhibo/.openclaw/runtime/openclaw-src-live/workspace-gateway-520239c5-b9b5-475e-b460-bc9f59571f95`

该目录中包含：

- `AGENTS.md`
- `SOUL.md`
- `TOOLS.md`
- `IDENTITY.md`
- `USER.md`
- `HEARTBEAT.md`
- `BOOTSTRAP.md`
- `.openclaw/workspace-state.json`
- `.git/`

这说明它不是静态源码目录，而是 OpenClaw 为某个 agent 实际创建并持续使用的运行 workspace。

## 2. 影响范围

- 运行态文件被写进源码树，污染 `runtime/openclaw-src-live`
- 源码目录和 agent 运行目录混在一起，后续排障、提交、升级都容易误操作
- 该目录一旦被旧 session 继续引用，哪怕主配置已改，也可能再次在源码树里被“种回来”

## 3. 根因分析

### 3.1 现场配置把 mc-gateway 的 workspace 直接写到了源码树里

在：

- `/Users/lizhibo/.openclaw/openclaw.json`

中，`mc-gateway-520239c5-b9b5-475e-b460-bc9f59571f95` 这个 agent 之前的 `workspace` 被显式配置为：

- `/Users/lizhibo/.openclaw/runtime/openclaw-src-live/workspace-gateway-520239c5-b9b5-475e-b460-bc9f59571f95`

因此 gateway 启动后会把这条路径当作真实工作区，并在其中写入 bootstrap 文件、heartbeat 文件和运行上下文。

### 3.2 源码里的非默认 agent workspace 回退路径也有隐患

在：

- `/Users/lizhibo/.openclaw/runtime/openclaw-src-live/src/agents/agent-scope.ts`

中，`resolveAgentWorkspaceDir(...)` 对非默认 agent 的隐式 workspace 原先是：

- `resolveStateDir(process.env) + /workspace-<agentId>`

这意味着只要 `stateDir` 被错误指向某个源码目录或运行目录，新的 agent workspace 也会继续落错地方。

文档里声明的语义本来就是：

- `main` 默认：`~/.openclaw/workspace`
- 其他 agent 默认：`~/.openclaw/workspace-<agentId>`

因此源码实现和文档语义并不完全一致。

### 3.3 旧 session 仍然把旧 workspace 路径固化在运行态

除了主配置以外，`mc-gateway` 现有 session 的运行元数据也保留了旧路径，例如：

- `/Users/lizhibo/.openclaw/agents/mc-gateway-520239c5-b9b5-475e-b460-bc9f59571f95/sessions/sessions.json`

中的 `systemPromptReport.workspaceDir`

以及 heartbeat transcript 中的提示文本，都仍然引用源码树里的旧目录。  
这会导致即使 `openclaw.json` 已经改正，旧 session 仍可能继续沿用旧 workspace。

## 4. 修复目标

本次修复分三层处理：

1. 修源码默认逻辑，避免以后非默认 agent 的隐式 workspace 再落到源码树
2. 修现场配置，把 `mc-gateway` 的显式 workspace 迁出源码目录
3. 清理旧 session 对旧路径的引用，避免源码树目录再次被重建

## 5. 实际处理过程

### 5.1 修复源码默认路径策略

修改文件：

- `/Users/lizhibo/.openclaw/runtime/openclaw-src-live/src/agents/agent-scope.ts`

调整内容：

- 新增 `resolveImplicitAgentWorkspaceDir(...)`
- 非默认 agent 的隐式 workspace 不再依赖 `resolveStateDir(process.env)`
- 改为固定落在默认 workspace 的父目录下，即：
  - `~/.openclaw/workspace-<agentId>`

这样即使未来某次运行环境里 `OPENCLAW_STATE_DIR` 被错误设置，也不会把 agent workspace 自动带进源码树。

### 5.2 补回归测试

修改文件：

- `/Users/lizhibo/.openclaw/runtime/openclaw-src-live/src/agents/agent-scope.test.ts`

新增测试覆盖：

- 当 `OPENCLAW_STATE_DIR` 被覆盖时，非默认 agent 的 workspace 仍然应该落在 `OPENCLAW_HOME/.openclaw/workspace-<agentId>`

### 5.3 修正 live 配置

修改文件：

- `/Users/lizhibo/.openclaw/openclaw.json`

将 `mc-gateway` 的 workspace 从源码树路径改为规范路径：

- 旧路径：
  - `/Users/lizhibo/.openclaw/runtime/openclaw-src-live/workspace-gateway-520239c5-b9b5-475e-b460-bc9f59571f95`
- 新路径：
  - `/Users/lizhibo/.openclaw/workspace-mc-gateway-520239c5-b9b5-475e-b460-bc9f59571f95`

配置变更前备份：

- `/Users/lizhibo/.openclaw/openclaw.json.workspace-relocation-20260307-2129.bak`

### 5.4 迁移实际 workspace

将原先位于源码树中的 workspace 正式迁出，保留原内容到新的标准位置：

- `/Users/lizhibo/.openclaw/workspace-mc-gateway-520239c5-b9b5-475e-b460-bc9f59571f95`

### 5.5 清理旧 session 粘连

由于 `mc-gateway` 的旧 session 已经把旧 workspace 路径写进 session 元数据和 heartbeat 历史中，因此仅改配置还不够。

处理方式：

- 备份原 session 目录：
  - `/Users/lizhibo/.openclaw/agents/mc-gateway-520239c5-b9b5-475e-b460-bc9f59571f95/sessions.workspace-relocation-20260307-2138.bak`
- 重建空的 session 目录：
  - `/Users/lizhibo/.openclaw/agents/mc-gateway-520239c5-b9b5-475e-b460-bc9f59571f95/sessions`

这样 `mc-gateway` 会在后续运行中基于新 workspace 重新建立干净 session，不再沿用旧路径历史。

### 5.6 将源码树里的旧目录移出源码范围

由于本机策略不允许直接做不可恢复删除，因此采用“迁出源码树”的方式处理旧目录。

旧目录被移到：

- `/Users/lizhibo/.openclaw/archive-generated-workspaces/workspace-gateway-520239c5-b9b5-475e-b460-bc9f59571f95.from-runtime-source-20260307-2138`

这意味着：

- `runtime/openclaw-src-live` 源码树中已经不再包含这个运行 workspace
- 如需回查现场，还可以到归档目录查看旧副本

### 5.7 重建并重启 gateway

基于 canonical 源码：

- `/Users/lizhibo/.openclaw/runtime/openclaw-src-live`

执行：

- `pnpm build`
- 重启 `ai.openclaw.gateway-src`

确保运行中的 gateway 使用的是修正后的源码和配置。

## 7. 最终结论

这次问题的本质不是“源码多了一个脏目录”，而是：

1. 配置把 agent workspace 误写进了源码树
2. 源码默认回退逻辑对非默认 agent 也存在把 workspace 带进错误根目录的风险
3. 旧 session 又进一步把错误路径固化，导致问题具备复发条件

本次处理后已经完成闭环：

1. 默认路径逻辑已修正
2. 现场配置已迁到正确位置
3. 旧 session 已备份并重置
4. 旧 workspace 已从源码树中移出
5. gateway 当前运行正常

## 8. 后续建议

- 以后新增 agent 时，优先显式指定 `workspace` 为 `~/.openclaw/workspace-<agentId>` 风格路径
- 避免把 runtime workspace 放在 `runtime/openclaw-src-live`、项目源码仓库或其他需要提交到 Git 的目录下
- 如果后续需要保留 `mc-gateway` 的 heartbeat 历史，可从：
  - `/Users/lizhibo/.openclaw/agents/mc-gateway-520239c5-b9b5-475e-b460-bc9f59571f95/sessions.workspace-relocation-20260307-2138.bak`
    中单独提取，不建议直接整目录覆盖回当前 live session
